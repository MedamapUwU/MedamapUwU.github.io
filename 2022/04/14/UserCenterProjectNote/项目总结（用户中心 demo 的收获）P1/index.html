项目总结（用户中心 demo 的收获）P1

流程

  1. 需求分析
  2. 对系统进行设计（概要设计，详细设计）
  3. 技术选型
  4. 初始化 / 引入技术（例如：Ant-Design、SpringBoot、umi）
  5. 写 Demo ：将引入的技术先跑通，保证能够在有基本数据的情况下运行起来
  6. 写代码：实现业务逻辑
  7. 测试：单元测试，写完一块功能，对这一部分功能进行单元测试 (Junit) ，不要等到所有的模块都完成后在测试，容易出现问题，早测出早解决
  8. 代码提交 / 代码评审
  9. 部署
  10. 发布

需求分析

即：我们要完成什么功能

  1. 最基本的 登录与注册
  2. 对登录的用户进行管理 （管理员权限）：对用户进行查询与修改
  3. 对注册用户进行校验（输入编号是否合法等）

技术选型

  • 前端
    • 三件套 (HTML + CSS + JavaScript) +
    • React
    • 组件库 Ant-Design
    • Umi
    • Ant-Design-Pro（现成的管理系统）
  • 后端
    • Java
    • Spring ：依赖注入框架，不需要自己管理 Java 对象，集成一些其他内容
    • SpringMVC ：Web 框架，提供接口访问，Restful 接口等
    • MyBatis ：Java 操作数据库的框架，即 持久层 框架，对 JDBC 的封装
    • MyBatis-Plus ：MyBatis 的增强版，提供了最基本的增删改查操作
    • SpringBoot ：不需要自己整合框架，可以快速启动项目
    • Junit 单元测试
    • MySQL 数据库

第一次接触 Ant-Design、Umi、Ant-Design-Pro 这种大型前端组件库，可以快速的帮助我们完成初始化，说实话还是挺震惊的，后端的 SpringBoot 框架也是快速初始化后端项目，但是没有前端这么华丽，相应的，前端的阅读起来会更加的费劲，因为没有太多的接触

数据库设计

User 表

字段 类型 备注
id bigint AUTO_INCREMENT PRIMARY Key
planetCode varchar(512) 星球编号
username varchar(256) 用户昵称
userAccount varchar(256) 用户账号
userPassword varchar(512) 用户密码
gender tinyint 性别
phone varchar(128) 手机号码
email varchar(128) 邮箱
avatarUrl varchar(1024) 用户头像
userRole int 用户角色 default 0
0 --> 普通用户
1 --> 管理员
userStauts int 用户状态 default 0
0 --> 正常
createTime datetime 创建时间 default CURRENT_TIMESTAMP
updateTime datetime 更新时间 default CURRENT_TIMESTAMP on update CURRENT_TIMESTAMP
isDelete tinyint 是否删除 0 --> 未删除 1 --> 删除

在这次做项目时我学到了 createTime,updateTime,isDelete 这三个新的字段，以前设计数据库表时从来没有意识到过这三点

isDelete 来表示逻辑删除，即并非将数据真的从数据库中删除，而是改变这条数据的状态，来让查询数据库时忽略这条数据，因为数据一旦删除，恢复起来就会很麻烦，因此删除应该从逻辑上删除，而非让这条数据从数据库中消失

这里使用了 MyBatis-X 插件，自动根据数据库生成 domain 实体对象，mapper (映射，即操作数据库的对象)，mapper.xml (定义了 mapper 对象和数据库字段之间的映射关系，可以在里面写 SQL 语句)、service（包含常用的增删改查）、serviceImpl （具体的 service 实现）

服务的逻辑 (Service 层)

注册

  1. 用户在前台输入：账号 userAccount，密码 userPassword，确认密码 checkPassword
  2. 校验输入的合法性
    1. 非空
    2. userAccount 长度不小于 4 位
    3. userPassword 长度不小于 8 位
    4. userAccount 不能重复
    5. userAccount 中不包含特殊字符
    6. userPassword 与 checkPassword 相同
  3. 通过后，对密码进行加密 （使用 SALT 值与 SpringBoot 自带的 md5 加密）密码一定不要以明文的形式存入数据库，保证安全性
  4. 向数据库插入数据

登录

接口

接收参数： 账号 userAccount ，密码 userPassword

请求方式： POST （因为请求体可能会很长，因此不建议使用 GET 请求方式，因为 POST 将信息放在请求体中，GET 放在请求头中）

请求体：JSON 格式的数据

返回值：用户信息（脱敏）

逻辑

  1. 用户在前台输入：账号 userAccount ，密码 userPassword
  2. 校验输入的合法性
    1. 非空
    2. userAccount 长度不小于 4 位
    3. userPassword 长度不小于 8 位
    4. userAccount 中不包含特殊字符
  3. 校验输入的密码执行与注册时相同的加密操作后得到的密文是否与数据库中的一致
  4. 对用户信息进行脱敏操作，隐藏敏感信息（例如密码），防止数据库中的字段泄露
  5. 记录用户的登录态 (Session)，将其存放到服务器上 (SpringBoot 自带了一个 Tomcat 服务器，因此用 Tomcat 去记录)，将用户登录态放入 Session 的 Attribute 域中
  6. 返回脱敏后的信息

注销

逻辑

  1. 通过 request 返回的信息查看用户是否存在
  2. 将 Session 的 Attribute 域中用户的 userStatus 移除

对于注册的密码一定不能将明文存入数据库中，保证数据的安全性。
