收获：

枚举类

因此，在比较两个枚举类型的值时，并不需要调用 equals ，只需要调用 == 即可

如果需要，可以为枚举类添加构造器，方法和字段，构造器只有在构造枚举常量的时候用得到

枚举的构造器总是私有的，可以省去 private 修饰符，如果声明构造器为 public 或者 protected,语法会出错

所有枚举类都是 Enum 的子类，其中最有用的方法为 toString 方法，他会返回枚举常量名。例如 ：Size.SMALL.toString() 将返回字符串 SMALL 。

toString 方法的逆方法时静态方法 valueOf，例如 ：Size s = Enum.valueOf(Size.class,"SMALL); 将 s 设置为 Size.SMALL

每个枚举类型都有一个**静态的 **values 方法，他将返回一个包含全部枚举值的数组。

ordinal 方法将返回 enum 声明中枚举常量的位置，位置从 0 开始计数。例如：Size.MEDIUM.ordinal() 将返回 1

继承使用的技巧

  1. 将公共操作和字段放在超类中
  2. 不要使用受保护的字段
  3. 使用继承实现 is-a 关系
  4. 除非所有继承的方法都有意义，否则不要使用继承
  5. 在覆盖方法时，不要改变预期的行为
  6. 使用多态，而不要使用类型信息
  7. 不要滥用反射

接口

接口的概念

接口中的所有方法都是 public 方法，因此，在接口中声明方法时，不必提供关键字 public ，不过在实现接口时，必须声明为 public

接口可能包含多种方法，但绝不会有实例字段。提供实例字段和方法实现的任务应由实现接口的那个类来完成

类实现一个接口，需要完成的两个步骤：

  1. 将类声明为实现给定的接口
  2. 对接口中的所有方法提供定义

以 Comparable 接口为例

接口的属性

接口不是类，具体来说，不能使用 new 来实例化一个接口

可以声明接口的变量：

接口变量必须引用实现了这个接口的对象

instanceof 既可以检查一个对象是否属于某个特定类一样，也可以检查一个对象是否实现了某个特定的接口

接口可以扩展，允许有多条接口链，从 通用 扩展到 专用

接口中不能包含实例字段，但是可以包含常量

与接口中的方法都自动设置为 public 一样，接口中的字段总是 public static final;

每个类只能有一个超类，但却可以实现多个接口。使用 , 将要实现的各个接口分隔开 。

默认方法

可以为接口方法提供一个默认实现，必须用 default 修饰符标记一个这样的方法

默认方法可以调用其他方法，例如 Collection 接口可以定义一个便利方法：

为接口增加一个非默认方法不能保证源代码兼容

解决默认方法冲突

如果先在接口中将一个方法定义为默认方法，然后又在超类或者另一个接口中定义相同的方法，如果解决冲突？

  1. 超类优先

    超类提供了一个方法，有同名同参的默认方法会被忽略

  2. 接口冲突

    如果一个接口提供了一个默认方法，另一个提供了一个同名而且参数类型（不论是否是默认参数）相同的方法，必须覆盖这个方法来解决冲突

类会继承两个接口的 getName() 的方法，并不是从中挑选一个, Java 编译会报告一个错误，让程序员来解决这个二义性的问题

如果其中一个接口不是默认实现，会自动继承那个默认实现的接口的 getName() 方法吗？答案是不会的，只要有冲突，就需要程序员解决这个二义性

如果两个接口都没有提供默认实现，那么就不会有冲突，那么就只有两个选择：一、实现这个方法；二、干脆不实现，那么这个类就必须是抽象类

以上是两个接口之间冲突

假如一个类扩展了一个超类，同时实现了一个接口，从超类和接口那继承了相同的方法，那么该如何解决冲突呢

这种情况下，我们只会考虑超类的方法，接口的所有默认方法都会被忽视。

类优先 规则可以确保与 Java7 的兼容性
